# Plan 02-01: Local-only guardrails implementation

## Objective

Implement Phase 2 guardrails per `02-CONTEXT.md` and `02-RESEARCH.md`: fail-closed outbound policy with documented allowlist and **local vs publish** lanes in `scripts/bump.sh`; vault path validation and safe JSON hook generation in `skills/code-to-docs-hooks/hooks/setup.sh` (and teardown alignment); a **pre-commit guard** plus developer-facing install docs; **minimal stdout hygiene** in hook scripts without Phase 3 redaction scope; and phase **VERIFICATION.md** with reproducible commands.

**Purpose:** Enforce local-first defaults, reduce accidental egress and accidental commits of sensitive artifacts, and preserve auditability (FR-4).

**Output:** Updated/new files listed in `files_modified`; traceability to D-01–D-10 where applicable.

**Task structure:** This plan uses **five sequential tasks in one wave** (not split across plans) because each task touches a different subsystem (release script, hook installer, git guard, stdout) and Phase 2 verification is intentionally reviewed as a single execution slice. **Phase 1 is complete** — baseline artifacts exist under `.planning/phases/1/` (PASS in `VERIFICATION.md`); this phase implements controls informed by that baseline.

**Allowlist enforcement model (D-02):** **In-repo policy documentation + gated `scripts/bump.sh`** — no separate “network firewall” wrapper process; unknown scripts are out of scope and not entitled to outbound I/O by default.

## Context

**Upstream baseline:** Phase 1 threat model / data flows / risk register are complete (`PASS` in `.planning/phases/1/VERIFICATION.md`). This plan implements guardrails; it does not re-baseline threats except where verification commands reference Phase 1 IDs.

## Trust boundaries

| Boundary | Description |
|----------|-------------|
| Maintainer script → network | `scripts/bump.sh` crosses from local disk to `git`/`gh`/GitHub. |
| Vault path → executable hook command | `setup.sh` embeds operator-supplied path into `.claude/settings.json` commands (DF-03 / TH-001). |
| Git index → remote | Pre-commit hook blocks sensitive paths before push (defense before egress). |

## STRIDE register (ASVS-oriented)

| Threat ID | Severity | Category | Component | Disposition | Mitigation (maps to task) |
|-----------|----------|----------|-----------|-------------|---------------------------|
| T-02-01 | **HIGH** | Tampering / Elevation | Hook command injection via `VAULT_PATH` or metacharacters in JSON shell fragment | mitigate | **Task 2:** Reject unsafe vault strings (D-05); build hook payloads with `json.dumps` from structured fields (D-05, V13 config integrity). Eliminates quote-breaking payloads reaching `settings.json`. |
| T-02-02 | MEDIUM | Information disclosure | Accidental egress via ambient `git push` / `gh release` | mitigate | **Task 1:** Local default lane; publish lane requires explicit env/flag beyond `read -rp` (D-03); remove silent push failure masking where inappropriate (D-01, D-08, V9). |
| T-02-03 | MEDIUM | Information disclosure | Hook stdout/transcripts echo full filesystem paths | mitigate | **Task 4:** Truncate or basename noisy paths; keep messages generic per D-09/D-10 (not full redaction library). |
| T-02-04 | MEDIUM | Integrity | Sensitive artifacts (`docs-vault`, `_state`, etc.) committed via `git add -f` | mitigate | **Task 3:** Pre-commit guard on cached paths aligned with `.gitignore` intent (D-06, D-07, V5 input validation). |

## Task 1 — Outbound policy document and `bump.sh` local vs publish lanes

**Files:** `docs/security/outbound-allowlist.md`, `scripts/bump.sh`

**Action:**

1. Add `docs/security/outbound-allowlist.md` documenting per **D-02**: (a) allowed outbound/network-capable scripts starting with `scripts/bump.sh`; (b) prerequisites (`gh` auth, network, repo identifier used by bump); (c) **D-03** publish-lane contract—document the exact env var and/or flag (e.g. `CODE_TO_DOCS_PUBLISH=1` or `--publish`) that must be set **in addition** to the interactive `Proceed?` prompt before any `git push` or `gh release create`; (d) default posture **fail-closed** per **D-01**, **D-08**; (e) **enforcement** = this allowlist **plus** the refactored `bump.sh` (no ambient network) — not a separate generic network-blocking wrapper; unknown scripts are not supported outbound entry points.
2. Refactor `scripts/bump.sh`:
   - **Local lane (default):** version bump, plugin.json update, optional commit, **local tag only** — no `git push`, no `gh`.
   - **Publish lane:** only when explicit publish gate is set per documented contract; then run push(es) and `gh release create`.
3. Remove `git push 2>/dev/null || true` (and similar) from publish paths—surface non-zero failures with **short, actionable** messages per **D-09**. Do not mask push failures in the publish lane.
4. If `gh` missing or `gh auth` fails when publish lane requested, exit non-zero fail-closed.
5. Reference decision IDs in comments where helpful (D-01–D-03).
6. Update `.planning/phases/02-local-only-guardrails-implementation/VERIFICATION.md` with **Task 1** rows: greps proving default lane has no `git push`/`gh release`; publish lane behind documented gate; link to `outbound-allowlist.md`.

**Verification:**

```bash
bash -n scripts/bump.sh
grep -E 'git push|gh release' scripts/bump.sh || true
# After implementation: document expected greps in VERIFICATION.md — publish-only blocks must be behind documented gate
```

**Done:**

- `docs/security/outbound-allowlist.md` exists and describes allowlist, prerequisites, and publish env/flag contract.
- Default invocation of `bump.sh` performs no remote operations.
- Publish lane requires documented second gate; push/gh failures are not silently swallowed.

## Task 2 — `setup.sh` vault validation and safe hook JSON (`json.dumps`)

**Files:** `skills/code-to-docs-hooks/hooks/setup.sh`, `skills/code-to-docs-hooks/hooks/teardown.sh` (only if shared helpers or consistent messaging are needed)

**Action:**

1. **D-05:** Before writing hooks, validate `VAULT_PATH` / `CODE_TO_DOCS_VAULT` resolution: reject empty path; reject characters that are unsafe in shell/JSON command context (quotes, newlines, control chars, and shell metacharacters per CONTEXT)—exit **1** with one remediation line per **D-09**.
2. **D-04:** Resolve repo root (`git rev-parse --show-toplevel` when inside a git worktree); if vault root is outside repo root, print a **high-signal warning** and continue.
3. Replace heredoc JSON assembly with **Python** building the hooks object and serializing with **`json.dumps`** so `command` strings are properly escaped (align with existing merge logic using `python3`).
4. Keep merge behavior with existing `.claude/settings.json` but ensure the fragment passed to merge is valid JSON from structured data (paths to `digest-on-start.sh`, `update-hint-on-commit.sh` as variables fed into Python).
5. **Teardown:** If `teardown.sh` shares path assumptions, ensure it does not re-expand unsafe paths; minimal change only if required for consistency.
6. Update `VERIFICATION.md` with **Task 2** checks: `bash -n` on edited hooks; documented negative vault path fixture exits non-zero; note JSON is built via structured data + `json.dumps`.

**Verification:**

```bash

bash -n skills/code-to-docs-hooks/hooks/teardown.sh
# Negative test (document in VERIFICATION.md): setup.sh with malicious vault string exits non-zero
```

**Done:**

- Unsafe vault paths fail closed with actionable errors.
- Out-of-repo vault emits warning per D-04.
- Hook JSON is emitted via `json.dumps`, not fragile heredoc embedding of `$VAULT_PATH`.

## Task 3 — Pre-commit guard script and install documentation

**Files:** `scripts/pre-commit-guard.sh`, `skills/code-to-docs-hooks/SKILL.md` (and optionally a short `skills/code-to-docs-hooks/hooks/README.md` if SKILL becomes too long—prefer extending SKILL.md to avoid extra files unless necessary)

**Action:**

1. Implement `scripts/pre-commit-guard.sh`: read `git diff --cached --name-only`; block commit if any path matches sensitive patterns per **D-06** and repo layout: at minimum `docs-vault/**`, `**/docs-vault/**`, `**/_state/**` or `**/_state/analysis.json`, and other paths consistent with `.gitignore` and Phase 1 data-flow sensitivity (align with `02-RESEARCH.md` baseline). Exit non-zero with **one** short line: what matched and how to unstage—per **D-09** (no secret dumps).
2. **D-08:** Fail-closed on policy violation.
3. Document installation: symlink or `core.hooksPath`, example `.git/hooks/pre-commit` snippet invoking repo-relative `scripts/pre-commit-guard.sh`; note defense-in-depth with `.gitignore` per **D-07**.
4. **Mandatory — `.gitignore` vs `scripts/`:** In `skills/code-to-docs-hooks/SKILL.md`, add one grep-friendly sentence: **ignore rules do not untrack already-tracked files** — `scripts/bump.sh` and `scripts/pre-commit-guard.sh` remain tracked maintainer scripts while `scripts/` still suppresses *new* untracked noise; pre-commit runs on staged paths regardless.

**Verification:**

**Done:**

- Executable guard script blocks sensitive staged paths per policy.
- SKILL.md (or hooks README) explains install steps and alignment with `.gitignore`.

## Task 4 — Minimal stdout hygiene in `digest-on-start.sh` and `update-hint-on-commit.sh`

**Files:** `skills/code-to-docs-hooks/hooks/digest-on-start.sh`, `skills/code-to-docs-hooks/hooks/update-hint-on-commit.sh`

**Action:**

Per **D-10**, apply **minimal** sanitization only (no Phase 3 redaction library):

1. `digest-on-start.sh`: Avoid printing full long absolute vault paths in routine messages—prefer basename or a truncated display for `$VAULT_PATH` in the “no vault” and summary lines; keep staleness/`git diff` output concise (e.g. limit verbosity of stat lines). Preserve existing summary behavior for project/modules/issue counts where low-risk.
2. `update-hint-on-commit.sh`: Confirm static/generic messaging; trim any path echo if introduced—keep output stable for automation.
